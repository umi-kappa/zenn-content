## はじめに

[前回の自己紹介記事](https://zenn.dev/umi_kappa/articles/9c717525720aa6)で「Claude Code を業務ではほぼ毎日使っています」と書きました。

あれから日々使い続ける中で、特にハマっているのが **Skill** という機能です。自分の業務に合わせたお手製のSkillをいくつか作って使っているのですが、これがなかなか便利で、チームにも展開し始めています。

この記事では、Skillとは何か、実際に業務で使っているSkillの紹介、作り方、そしてチームへの展開フローについて書いていきます。

## Claude CodeのSkillとは？

Skillは、Claude Codeに特定のタスクの手順やドメイン知識を教えておける仕組みです。要するに **Claude Codeへのカンペ** みたいなものです。

Markdownファイル（`SKILL.md`）として配置しておくと、Claude Codeがタスクに応じて自動的に適切なSkillを参照してくれます。

```
skill-name/
├── SKILL.md          # Skill本体（必須）
├── scripts/          # 実行スクリプト（任意）
└── references/       # 参照ドキュメント（任意）
```

配置場所は2つあり、用途が異なります。

- **User-level**（`~/.claude/skills/`）: 自分専用。どのプロジェクトでも使える
- **Project-level**（`プロジェクト/.claude/skills/`）: チーム共有。リポジトリにcommitして全員が使える

## 使っているSkillを紹介

現在使用しているSkillをカテゴリに分けて紹介します。

### コード生成系

#### API関連ファイル生成Skill

新しいAPIエンドポイントを追加するとき、「`〜のAPIを追加して`」と伝えるだけで、各レイヤーのファイルの **一括生成+既存ファイルの編集** を自動でやってくれます。新しいファイルを作るだけでなく、既存のコードに登録・接続する編集まで自動化されているのがポイントです。テストやモックデータのビルダーまで含めて生成されます。

手作業だと30分以上かかっていた定型作業が、数分で終わるようになりました。

#### ページコンポーネント生成Skill

業務のプロジェクトではモダン環境（Vue 3）とレガシー環境（Vue 2）が共存するデュアルアーキテクチャになっています。新規ページを作るとき、両環境のファイル生成からルーティング設定、Web Component登録まで **一括生成+既存ファイルの編集** を一発でやってくれます。

レガシーとモダンの橋渡しの設定を手でやると地味に漏れやすいので、Skillに任せられるのは安心感があります。

#### UIコンポーネント生成Skill

Vue 3コンポーネントとStorybookのファイルを自動生成します。FigmaのURLを渡すとデザインを読み取ってコードに落としてくれるので、デザインからの実装がスムーズになりました。

### ワークフロー系

#### PRレビュー用Worktree作成Skill（add-pr-worktree）

PRのレビューを始めるとき、PR番号やURLを渡すだけでレビュー用のworktreeを作成し、依存パッケージのインストールまで自動で完了します。

```bash
/add-pr-worktree 123
# or
/add-pr-worktree https://github.com/owner/repo/pull/123
```

レビュー環境のセットアップが一瞬で終わるので、「レビューするか」の心理的ハードルが下がりました。さらに、Worktree作成後はレビュー用カスタムエージェントと組み合わせて使っています。カスタムエージェントについては、いろいろ試行錯誤した点があるので、別の記事で詳しく紹介する予定です。

#### Worktree削除Skill（remove-worktree）

レビューが終わったら「レビュー完了」と言うだけでworktreeをクリーンアップして元のリポジトリに戻れます。未コミットの変更がある場合は確認してくれるので、うっかり作業中のコードを消してしまう心配もありません。

add-pr-worktreeとremove-worktreeはセットで使っていて、PRレビューのワークフローがかなり快適になりました。

#### リリース前QA Issue作成Skill

リリース前にQAチームへ依頼するためのIssueを自動作成するSkillです。MilestoneからIssueを収集し、追加機能・不具合改修・開発環境改善などに自動分類して、QAチーム向けのまとめIssueを作成、Projectボードへの登録までやってくれます。

地味だけど大事なリリース前の定型作業。手作業だと分類漏れが起きがちでしたが、Skillに任せてからはそれがなくなりました。

## Skillの作り方

Skillを作るのは実はそこまで難しくありません。Claude Codeには **[skill-creator](https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md)** という「Skillを作るためのSkill」がAnthropic公式で用意されています。インストール方法や詳しい使い方は、以下の記事が参考になります。

- [Agent Skillsを一番かんたんに作る方法（Claude Code + skill-creator）](https://zenn.dev/aun_phonogram/articles/475f3cca8f40a3)
- [Anthropic公式「skill-creator」完全ガイド](https://zenn.dev/yamato_snow/articles/30e8dcf9490c88)

作り方の流れはざっくりこんな感じです。

1. やりたいタスクをClaude Codeに説明する
2. skill-creatorが `SKILL.md` のドラフトを生成してくれる
3. 実際に使ってみて、うまくいかない部分をAIと相談しながら調整する

最初から完璧なSkillを目指す必要はなく、 **使いながら育てていく** 感覚が大事だと思っています。何度か使ってみて「ここはもう少しこうしてほしい」と思ったら、その都度AIと相談して `SKILL.md` を調整していけばOKです。

## 業務でProjectにお手製Skillをpushするまでの流れ

自分がSkillをチームに展開するまでの流れを紹介します。

### Step 1: User-levelでまず作って試す

まずは `~/.claude/skills/` にSkillを作って、自分だけの環境で試行錯誤します。この段階では雑でOK。「とりあえず動く」レベルで十分です。

実際に業務で数回使ってみて、「このSkillのこの手順、毎回微妙にずれるな」とか「この情報も教えておいた方がいいな」という気づきを拾って調整していきます。

### Step 2: チームに共有して意見をもらう

ある程度使えるレベルになったら、使い方と合わせてチームに共有します。「こういうSkill作ったんですけど、使ってみませんか？」という感じです。

実際に他のメンバーに使ってもらうと、自分では気づかなかった改善点が見つかることも多いです。フィードバックをもらったら取り込んで改善します。

### Step 3: Project-levelに移動してpush

チームで使えるレベルに仕上がったら、`~/.claude/skills/` から `プロジェクト/.claude/skills/` に移動してリポジトリにcommitします。これでチーム全員が同じSkillを使える状態になります。

## 今後の展望

- まだUser-levelにあるSkillを育て、Project-levelへ移動し、チーム全体の開発効率を上げたい
- 効率的に動く（Tokenが少ない）Skillを作れるようになりたい

Skillは **育てるもの** だと思っています。業務の変化に合わせて、使い続けながら少しずつ良くしていく。その積み重ねが、チーム全体の開発体験を変えてくれると信じています。

## おわりに

Skillを使い始めてから、定型作業にかかる時間と心理的な負担がかなり減り、開発に使える時間が少しずつ増えてきました。「あの面倒な手順、Skillにしちゃおう」と思える感覚が身についたのも大きいです。

とはいえ、まだまだSkill内部の精度には自信がないので、そのあたりもトライアンドエラーを繰り返しながら会得していければと思います！
